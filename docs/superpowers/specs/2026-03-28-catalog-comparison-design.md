# Stitch Showcase — Integrated Component Catalog & Comparison

## Context

Google Stitch exports generate slight variations of shared components (navbar, footer, buttons) across screens. The current `--catalog` flag produces a component inventory, but:

1. **Previews are broken**: Components render as unstyled HTML because the Tailwind CDN + config from the original screen HTML is stripped during extraction
2. **Catalog is optional**: Users must explicitly request `--catalog` instead of getting it automatically
3. **No comparison/selection**: Users can see variants exist but can't compare them visually or select a canonical version to standardize across screens

This spec addresses all three problems in a unified design.

## Architecture

### 3-Page Showcase (always generated together)

| Page           | Purpose                                                 |
| -------------- | ------------------------------------------------------- |
| `index.html`   | Screen gallery + Manual de Identidad + link to catalog  |
| `viewer.html`  | Individual screen viewer (phone frame / browser chrome) |
| `catalog.html` | Component inventory with comparison & selection         |

All three pages are generated by a single `build_showcase.py` run. The `--catalog` and `--components` flags are removed — catalog generation is always included.

### Data Flow

```
Screen HTMLs (assets/*.html)
    │
    ├── extract_catalog.py ──────► component_catalog.json (atomic + composite)
    │                               + tailwind config extraction
    ├── detect_components.py ────► shared_components.json (structural clusters)
    │
    └── build_showcase.py ───────► index.html + viewer.html + catalog.html
                                      │
                                      └── catalog.html consumes both JSONs
                                           + tailwind CDN/config for previews
```

## Part 1: CSS Extraction for Faithful Previews

### Problem

Stitch HTMLs use Tailwind CDN with a custom `tailwind.config` defining project colors (`primary`, `surface`, `tertiary`, etc.). When `extract_catalog.py` strips `<style>` and `<script>` blocks via `_strip_invisible()`, the extracted component snippets lose all styling.

### Solution

Extract two elements from the first screen HTML (all screens share the same config):

1. **Tailwind CDN script tag**: `<script src="https://cdn.tailwindcss.com?plugins=...">`
2. **Tailwind config script**: `<script id="tailwind-config">tailwind.config = {...}</script>`

New function in `extract_catalog.py`:

```python
def extract_tailwind_head(assets_dir: Path) -> str:
    """Extract Tailwind CDN + config from first screen HTML.
    Returns HTML string to inject in <head>."""
```

### Template Changes

- `catalog-template.html`: Add `{{TAILWIND_HEAD}}` placeholder in `<head>`
- Preview wrappers: Add `class="dark"` when project uses dark mode
- Component previews now render with full Tailwind styling

### Files Modified

- `scripts/extract_catalog.py` — add `extract_tailwind_head()`
- `scripts/build_showcase.py` — inject tailwind head into catalog template
- `references/catalog-template.html` — add `{{TAILWIND_HEAD}}` placeholder

### Variant Display Logic

For each component type:
1. If only 1 variant exists across all screens → show in "Already Unified" section (collapsed)
2. If multiple clusters exist → show side-by-side cards per cluster
3. Each card shows:
   - Styled preview (via Tailwind extraction)
   - Canonical badge (★) on the auto-detected best version
   - Similarity score relative to canonical
   - Screen count + screen name list
   - Difference description (e.g., "Missing text: Dashboard", "Different background color")

### Cluster Logic

**Structural components** (navbar, footer, sidebar, tabbar):
- Clustered by `detect_components.py` at 85% similarity threshold
- Canonical auto-selected by priority: home-like slug → most DOM nodes → first found
- Replacement is global: chosen variant replaces in ALL screens

**Atomic components** (buttons, inputs, badges, headings):
- Clustered by `extract_catalog.py` using HTML hash (exact structure match)
- Extended with similarity-based clustering for near-matches (new logic needed)
- Context-aware: clusters respect parent context. Context is determined by the nearest semantic ancestor (`<form>`, `<section>`, `<header>`, `<footer>`, `<nav>`, `<aside>`) or, if none, by position in the DOM (top/middle/bottom third of the page). Two buttons in `<form>` elements are same-context; a button in a `<form>` vs one in a `<section class="hero">` are different contexts.
- Replacement is per-cluster: only affects screens within that cluster

### How Selection Works

The catalog page is **static HTML** — it has no backend or interactivity for selection. The user:

1. Opens `catalog.html` in the browser
2. Sees all component clusters with their variants side-by-side
3. Tells the AI agent which variant to use: "usa el navbar del home" or "usa el boton primario para los formularios"
4. The AI agent runs `apply_canonical.py` to replace variants in the source HTMLs
5. The AI agent regenerates the showcase (which now includes the updated catalog)
6. Repeat until the catalog shows everything as "Already Unified"

### New Script: `apply_canonical.py`

```python
def apply_canonical(
    assets_dir: Path,
    component_type: str,      # "navbar", "footer", "button", etc.
    canonical_slug: str,       # screen slug to use as source
    target_slugs: list = None  # if None, apply to all screens with this component
) -> dict:
    """Replace component variants in target HTMLs with the canonical version.

    Returns dict with {slug: "replaced" | "skipped" | "not_found"} per screen.
    """
```

**Replacement strategy for structural components**:
1. Extract the component block from canonical screen HTML (using `component_utils.extract_semantic_blocks()`)
2. For each target screen, find the matching component block
3. Replace the block in-place
4. Write updated HTML back to disk

**Replacement strategy for atomic components**:
1. Use the exact HTML snippet from the canonical
2. Find matching snippets in target screens (by cluster membership)
3. Replace only within the same context/cluster
4. Write updated HTML back to disk
